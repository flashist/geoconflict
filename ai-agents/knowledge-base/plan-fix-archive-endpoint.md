# Plan: Fix Archive Endpoint Failures and Singleplayer Body Limit

> **Status:** Decision recorded 2026-06-01. **Deferred** — the correct fix is S3-backed
> archival, but the underlying feature (game history) is gated to "citizen" users, and
> citizenship is not implemented yet, so **no one will have archiving/history in the near
> future**. For now we only **reduce the telemetry noise**; the real S3 implementation is a
> later task. The earlier disk-on-master approach in this file has been **superseded** (see
> _Rejected alternative_).

---

## Context

Three production error groups (2026-05-07 Uptrace window, ~26.6/min total) are generated by
the archive path even though nothing currently consumes the archived data:

| Error | Rate | Failing leg |
|---|---|---|
| `error archiving game record (...): Not Found` | 16.29/min | server → archive backend |
| `TypeError: Failed to archive singleplayer game: Failed to fetch` | 9.64/min | client → worker |
| `PayloadTooLargeError: request entity too large` on `/api/archive_singleplayer_game` | 0.64/min | client → worker |

Source: `ai-agents/knowledge-base/telemetry-error-priorities-2026-05-07.md`

---

## Root Cause Analysis (verified against the code — keep for the future implementer)

### Issue 1 — Archive 404 (`Archive.ts`), affects BOTH game types
`archive()` POSTs to `${config.jwtIssuer()}/game/:id`. In prod `jwtIssuer()` resolves to
`http://91.197.98.116` (the game host itself), which has **no `/game/:id` route** → 404.
`archive()` is shared by **multiplayer** (`GameServer.ts:992`) and **singleplayer**
(`Worker.ts:269`), so a single change governs both server-side legs.

Upstream openfront.io had an external Cloudflare Worker at `api.openfront.io/game/:id`. The
geoconflict fork inherited the archive *client* but never stood up a matching endpoint.
Confirmed prod env: `JWT_ISSUER=http://91.197.98.116`, `API_BASE_URL=http://91.197.98.116`.

### Issue 2 — Singleplayer `Failed to fetch` (`LocalServer.ts`)
`LocalServer.endGame()` sends the gzip POST with `keepalive: true`. Browsers cap keepalive
request bodies at **64 KB**; larger compressed records throw `TypeError: Failed to fetch`
**before the request leaves the client**. This is why client failures (9.64/min) dwarf the
server-side 413s (0.64/min).

### Issue 3 — Singleplayer 413 (`Worker.ts`)
The route is covered by the global default `express.json()` (100 KB). For a **gzip** request
body-parser leaves `stream.length` unset, so the limit is enforced on the **decompressed**
size (verified in `node_modules/body-parser/lib/read.js`). A record that slips under the
64 KB keepalive cap can still decompress past 100 KB and 413. (Note: an earlier draft claimed
the limit applies to the *compressed* size — that is wrong.)

## Decisions (2026-06-01)

1. **S3-compatible object storage is the correct destination.** `DefaultConfig` already
   exposes `storageEndpoint()` / `storageBucket()` / `storageAccessKey()` /
   `storageSecretKey()` (currently unused). That scaffolding is the architecturally-intended
   archive sink. Archival should write there — "use the bucket the way the project supposes."

2. **Archiving/history is a "citizen"-user feature, and citizenship is not implemented yet.**
   In practice nobody has (or will soon have) the history feature, so there is **no live
   consumer** of archived records. We are paying the noisiest archive error group in
   telemetry to populate a store nothing reads.

3. **Disk-on-master is rejected** (see _Rejected alternative_): unbounded disk growth / no
   retention, and it diverges from the intended S3 architecture.

4. **Two phases:** reduce telemetry noise now (no storage built); implement S3 archival,
   gated to citizen games, when the citizenship feature lands.

---

## Phase 1 — Interim: reduce telemetry noise (do now)

Goal: drop all three error groups to ~0 **without building any storage**, since archival is
intentionally inactive. The lightest sufficient change is to stop the doomed calls and/or
demote their logs.

- **Server `archive()` (`src/server/Archive.ts`)** — short-circuit (early return / no-op)
  so it no longer POSTs to the non-existent endpoint, OR at minimum demote the failure
  `log.error` to `log.warn`/remove it. Short-circuiting is preferred: it also stops the
  wasted network call and removes the 16.29/min group for both multiplayer and singleplayer
  server legs. Keep the function structure so re-enabling in Phase 2 is a one-line gate flip.
- **Singleplayer client (`src/client/LocalServer.ts`, `endGame()`)** — while archival is
  inactive, skip the upload (don't `fetch` the compressed record), OR at minimum change the
  catch `console.error` → `console.warn`. Skipping removes both the 9.64/min `Failed to
  fetch` and the 0.64/min 413 and avoids uploading data the server discards.
- Do **not** raise body limits, add Master routes, or write to disk in this phase.

**Acceptance:** the three Uptrace groups disappear/drop sharply; no archive storage exists;
the change is a small, clearly-marked "feature temporarily disabled" edit.

---

## Phase 2 — Future: S3 archival, citizen-gated (when citizenship ships)

1. **Implement S3 write in `archive()`** using the existing `storageEndpoint()` /
   `storageBucket()` / `storageAccessKey()` / `storageSecretKey()` config. Store each record
   as an object keyed by gameID (e.g. `game-records/<gameID>.json`). Add the read counterpart
   for replays (replacing the current `readGameRecord()` GET).
2. **Gate archival to citizen games only** — do not archive every game. Multiplayer: derive
   citizen players from JWT claims/roles in `GameServer`. Singleplayer: gate the client
   upload on the user's citizen status before POSTing. This both honours the product rule and
   slashes volume/cost vs. archiving everything.
3. **Re-enable** the singleplayer client upload and the server archive path behind the same
   gate flipped on in Phase 1.
4. **Body limit:** when uploads resume, set an explicit, bounded `express.json` limit on the
   singleplayer ingest route — remember it is enforced on the **decompressed** size for gzip.
   Validate against real record sizes; do not use an unbounded limit.
5. **Keep richer archive error logging** (status code + URL + short safe body excerpt) for
   diagnosability — it stays dormant while disabled and pays off once S3 is live.

---

## Rejected alternative — disk-on-master (do not re-propose)

Mirror the cosmetics fix: add `POST/GET /game/:id` to `Master.ts`, persist records as JSON
files under `data/game-records/`, point `archive()` at `localMasterUrl()`. **Rejected**
because: (a) unbounded disk growth with no retention on a single VPS (multi-MB records ×
~16 game-ends/min), and (b) it diverges from the S3 architecture the project already provides
config for. It also would have required raising `express.json` limits on **both** Worker and
Master (multiplayer records reach Master uncompressed) plus auth + rate-limit exemptions —
extra surface for a store we shouldn't be building.

## Verification

**Phase 1:** after deploy, the `Not Found`, `Failed to fetch`, and `PayloadTooLargeError`
archive groups fall to ~0 in Uptrace; no `data/` directory or new routes are introduced.

**Phase 2:** citizen multiplayer/singleplayer games produce objects in the S3 bucket;
non-citizen games are not archived; replays read back from S3; oversized uploads fail with a
clean bounded log; non-citizen flows produce no archive errors.
