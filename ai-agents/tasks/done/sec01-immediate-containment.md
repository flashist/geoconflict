# Task — Immediate Containment And Secret Rotation

## Type
Urgent incident-response runbook.

## Purpose

Contain the incident before deeper investigation by rotating or invalidating every secret that could have been exposed through `.env`, `.env.prod`, the VPS, or a pushed Docker image.

## Why This Matters

If the attacker copied `VPS_*` from a leaked env file, they may also have seen every other secret stored alongside it. Containment must happen before root-cause analysis or repo cleanup.

## Inputs Required

- Current contents of `.env`
- Current contents of `.env.prod`
- Access to the VPS provider, registry, telemetry, storage, Telegram/webhook providers, and any admin backends
- Access to the deploy operator machine that normally runs `build.sh` / `deploy.sh`

## Preconditions

- Stop normal deployments until containment completes
- Assume all secrets in the current env files are compromised
- Create a temporary secure note or spreadsheet to track rotations, owners, and timestamps
- Use `scripts/list-secret-keys.sh` to inventory variable names without printing secret values

## Repo-Specific Secret Surface

The following secret classes are confirmed from the current repo and must be considered during containment.

### Core game deploy secrets (`.env`, `.env.<env>`, `deploy.sh`)

| Secret class | Variables / sources | Why it matters |
|---|---|---|
| VPS access | `VPS_IP`, `VPS_LOGIN`, `VPS_PASSWORD`, optional `SSH_USER_<ENV>`, `SSH_PASS_<ENV>`, `SSH_KEY` | Direct host access and deployment auth |
| Registry / build | `DOCKER_TOKEN`, `DOCKER_USERNAME`, optional shell-only `SENTRY_AUTH_TOKEN` passed by `build.sh` | Image push and build-time auth |
| Game admin / API | `ADMIN_TOKEN`, `API_KEY` | Server-side administrative and API trust |
| Storage | `STORAGE_ENDPOINT`, `STORAGE_ACCESS_KEY`, `STORAGE_SECRET_KEY`, `STORAGE_BUCKET` | Object storage access and data integrity |
| Game telemetry | `OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_AUTH_HEADER`, `OTEL_USERNAME`, `OTEL_PASSWORD`, `OTEL_ENDPOINT` | Production telemetry pipeline and auth |
| Feedback / messaging | `FEEDBACK_WEBHOOK_URL`, `FEEDBACK_TELEGRAM_TOKEN`, `FEEDBACK_TELEGRAM_CHAT_ID`, `TELEGRAM_PROXY_URL` | Webhook and Telegram delivery paths |
| Basic auth | `BASIC_AUTH_USER`, `BASIC_AUTH_PASS` | Staging or prod access gate if enabled |

### Telemetry-side secrets (`.env.telemetry`, telemetry scripts)

| Secret class | Variables / sources | Why it matters |
|---|---|---|
| Telemetry VPS access | `TELEMETRY_SERVER_HOST`, `TELEMETRY_VPS_LOGIN`, `TELEMETRY_VPS_PASSWORD`, `TELEMETRY_SSH_KEY` | Direct access to the Uptrace host |
| Uptrace project auth | `UPTRACE_PROJECT_TOKEN`, `UPTRACE_SECRET_KEY`, `UPTRACE_ADMIN_PASSWORD` | Dashboard access and OTLP ingest trust |

### Runtime copies created by `deploy.sh`

These values are explicitly written into the temporary remote env file during deploy and should be treated as runtime secrets on the VPS:

- `DOCKER_TOKEN`
- `ADMIN_TOKEN`
- `API_KEY`
- `STORAGE_*`
- `OTEL_*`
- `BASIC_AUTH_*`
- `FEEDBACK_*`
- `TELEGRAM_PROXY_URL`

`deploy.sh` does **not** copy `VPS_PASSWORD` to the runtime env file, but the presence of leaked `VPS_*` values means the local operator env files or a built image are still in scope.

## Actions

### 1. Inventory everything that must be rotated

From `.env`, `.env.prod`, `.env.telemetry`, shell exports, and any adjacent operator-only files, build the master list.

Start with a safe key-name-only inventory:

```bash
./scripts/list-secret-keys.sh .env .env.prod .env.telemetry
```

If a file does not exist locally, note that explicitly in the rotation sheet instead of assuming it was never used.

At minimum, include:

- `VPS_PASSWORD`
- any SSH private keys used for deploys or copied to the host
- `DOCKER_TOKEN`
- `ADMIN_TOKEN`
- `API_KEY`
- `STORAGE_ENDPOINT`, `STORAGE_ACCESS_KEY`, `STORAGE_SECRET_KEY`, `STORAGE_BUCKET`
- `OTEL_USERNAME`, `OTEL_PASSWORD`, `OTEL_ENDPOINT`, `OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_AUTH_HEADER`
- `FEEDBACK_WEBHOOK_URL`
- `FEEDBACK_TELEGRAM_TOKEN`
- `FEEDBACK_TELEGRAM_CHAT_ID`
- `TELEGRAM_PROXY_URL`
- any secrets found in `.env.telemetry` or provider dashboards

Do not skip items just because they look low risk.

Also include build-operator secrets that may not live in tracked example files:

- `SENTRY_AUTH_TOKEN` if present in shell config or CI
- any Docker credentials stored via `docker login`
- provider-side tokens generated by telemetry bootstrap scripts

Use [docs/security/secret-rotation-inventory.template.md](/Users/mark.dolbyrev/Workspace/geoconflict/docs/security/secret-rotation-inventory.template.md:1) as the blank structure, but fill the real data in a private copy outside the repo.

### 2. Rotate the SSH and VPS access layer first

- Change the VPS password immediately
- If password auth is currently enabled, disable it after a key-based path is confirmed
- Review all authorized SSH keys and remove unknown ones
- If trust in current keys is low, replace them as well

### 3. Rotate external service tokens

Rotate all provider-side credentials that appear in env files:

- Docker registry token
- storage credentials
- OTEL or Uptrace auth
- webhook and Telegram credentials
- app admin tokens and API keys

Where rotation is impossible, invalidate the old credential and create a replacement.

Rotation order for this repo should be:

1. production VPS access
2. Docker registry credentials
3. production runtime secrets copied by `deploy.sh`
4. telemetry / Uptrace credentials
5. feedback / webhook / Telegram credentials
6. any remaining build-only operator credentials

### 4. Replace local operator env files

- Generate clean replacement values
- Update local operator env files only after provider-side rotation is complete
- Mark old env files as obsolete and archive them securely if needed for forensic reference

For this repo specifically:

- replace `.env`, `.env.prod`, and `.env.telemetry` only after upstream rotations are complete
- do not keep obsolete env files inside the repo workspace
- if forensic preservation is needed, move obsolete copies to a private non-repo location with restricted permissions

### 5. Freeze unsafe deploy paths

Until `sec04` and `sec05` are complete:

- do not run `build.sh` from a workstation that still has old secrets
- do not deploy from old images or old tags
- do not use password-based SSH if a key-based path exists
- do not run `build-deploy-telemetry.sh` with pre-rotation telemetry credentials
- do not use `npm run telemetry:tunnel` if it still depends on `TELEMETRY_VPS_PASSWORD`

## Fast Checklist

Use this as the minimum incident containment pass:

1. Freeze deploys and telemetry redeploys.
2. Inventory `.env`, `.env.prod`, `.env.telemetry`, and shell-only secrets.
3. Rotate prod VPS password and/or migrate to key-only SSH.
4. Rotate Docker registry credentials.
5. Rotate all runtime secrets written by `deploy.sh`.
6. Rotate Uptrace / telemetry VPS credentials.
7. Replace local operator env files with new values.
8. Verify old credentials fail.
9. Keep old images and old tags quarantined for `sec02`.

## Evidence To Collect

- Rotation sheet with secret name, owner, rotation timestamp, and status
- Confirmation that old credentials fail
- Screenshot or command output proving password SSH is disabled, if completed here
- Notes on any credentials that could not yet be rotated and why
- Output of `scripts/list-secret-keys.sh` captured privately for the real env files
- A note showing which deploy/operator files were replaced locally after rotation

## Done Criteria

- Every secret in `.env` and `.env.prod` is either rotated or explicitly listed as blocked with owner and ETA
- Every secret in `.env.telemetry` and shell-only deploy configuration is either rotated or explicitly listed as blocked
- The old VPS password no longer works
- Deploy operators are using only new secret values
- Unsafe deploy activity is paused until the clean rebuild path is ready

## If Blocked

- If a provider credential cannot be rotated immediately, disable or restrict the integration if possible
- If SSH key-only access cannot be enabled immediately, reduce access by IP or firewall rules while the VPS audit proceeds
- If the host trust level is too low, treat replacement of the VPS as a valid containment path

## Outputs For Next Steps

- Clean secret inventory for `sec06`
- Confirmed list of deprecated credentials to verify as invalid during validation
